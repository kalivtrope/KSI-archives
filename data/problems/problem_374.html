Zadání

Pytest je framework pro psaní a spouštění testů v Pythonu.

Jak to funguje si ukážeme na příkladu algoritmu pro výpočet 2x2x, který jsme už použili v nulté vlně. Zapsaný v Pythonu vypadá takto:

def pow_two(x: int) -> int:
    r = 1
    i = 0
    while i != x:
        r = r * 2
        i = i + 1
    return r

No a tady je k němu test zapsaný v Pytestu. Pytestové testy jsou funkce, jejichž jméno začíná na test_.

def test_pow_two() -> None:
    assert pow_two(0) == 1
    assert pow_two(1) == 2
    assert pow_two(2) == 4
    assert pow_two(3) == 8
    assert pow_two(4) == 16
    assert pow_two(5) == 32

Funkce pow_two a test_pow_two jsme napsali do souboru, který se v této ukázce jmenuje pow_two.py.

Abychom mohli test spustit, musíme nejprve nainstalovat Pytest:

python -m pip install pytest

Test spustíme pomocí:

python -m pytest pow_two.py

Právě uvedený příkaz postupně provede všechny funkce v souboru pow_two.py, jejichž jméno začíná na test_. Pokud některý assert selže, Pytest ukončí provádění aktuální testové funkce, a spustí další testovou funkci v souboru. Na konci vypíše Pytest obsáhlou chybovou zprávu pro každý test, který selhal.

(Občas má smysl napsat selhávající test. Zejména pokud test zachycuje něco, co mělo fungovat a nefunguje, nebo nikdy nefungovalo, a ty se chystáš vzápětí upravit program tak, aby test prošel. Prvním krokem v přístupu nazvaném Test-first Development je napsat (selhávající) testy, které testují to, co se chystáš naprogramovat. Mělo by to vést k tomu, že si při psaní testů ujasníš, co vlastně chceš vytvořit, a co by to mělo dělat.)

V porovnání s jednoduchými asserty bez Pytestu jsme použitím tohoto frameworku získali následující:

  • podrobné chybové hlášky pro selhávající asserty
  • selhávající assert ukončí jen aktuální testovací funkci a Pytest bude pokračovat s další testovací funkcí
  • test discovery; Pytest si umí najít všechny testové funkce v našem projektu a spustit je
  • podpora pro knihovnu unittest; pokud máš v projektu testy napsané s touto knihovnou, Pytest je taky najde a pustí
  • přístup k další pokročilé Pytestové funkcionalitě, jako jsou parametrizované testy, pluginy na HTML reporty, pokrytí kódu, a tak podobně

Jeden test, jeden assert (Parametrizované testy)

The builtin pytest.mark.parametrize decorator enables parametrization of arguments for a test function.

Parametrizing fixtures and test functions (docs.pytest.org)

Častá začátečnická rada pro psaní testů zní, že každý test by měl assertovat jen jednou. Tím pádem pokud test selže, znamená to, že v testovaném kódu nefunguje jedna konkrétní věc. Ideálně je už ze jména testové funkce jasné, která.

Pokud potřebuješ assertů víc, máš napsat víc testů. Je to dobrá rada, protože lidé mají tendenci psát dlouhé testové funkce, ve kterých je obtížné se vyznat.

Kdybychom chtěli radu aplikovat na náš test, dostaneme

def test_pow_two_0() -> None:
    assert pow_two(0) == 1


def test_pow_two_1() -> None:
    assert pow_two(1) == 2

...

Abychom se vyhnuli psaní repetitivních funkcí tohoto typu, použijeme funkcionalitu pro parametrizované testy, kterou Pytest nabízí. Funguje to tak, že definujeme testovou funkci s jedním nebo více parametry a aplikujeme na ni dekorátor, kterým Pytestu řekneme, s jakými argumenty by měl při testování funkci zavolat.

import pytest


@pytest.mark.parametrize("x,expected", [
    (0, 1),
    (1, 2),
    (2, 4),
    (3, 8),
    (4, 16),
    (5, 32),
])
def test_pow_two(x: int, expected: int) -> None:
    assert pow_two(x) == expected

Mít testové případy v testu explicitně vypsané má své výhody, zejména že je na první pohled vidět, co od funkce pow_two očekáváme. Pokud máme testovacích případů hodně, například pokud chceme vyčerpávajícím způsobem otestovat všechny realistické vstupy, se kterými se náš program v praxi setká, je lepší zvolit jiný přístup:

import pytest


def generate_pow_two_cases():
    for x in range(6):
        yield x, 2**x


@pytest.mark.parametrize("x,expected", generate_pow_two_cases())
def test_pow_two(x: int, expected: int) -> None:
    assert pow_two(x) == expected

Právě ukázané funguje proto, že dekorátor @pytest.mark.parametrize přijímá jako druhý argument libovolnou iterable hodnotu, takže můžeme použít nejen třeba list, ale i generátor.

Generátorová funkce generate_pow_two_cases by mohla číst předchystané testovací vstupy ze souboru, třeba z Excelovské tabulky, a tak podobně, ale to si zde ukazovat nebudeme.

Terminologická vsuvka: knihovny a frameworky

Softwarové knihovny jsou seskupení znovupoužitelných funkcí, tříd a modulů, určené pro použití ve více než jednom programu. V Pythonovém světě je organizujeme do balíčků a instalujeme pomocí pipu, se kterým už jsme se v KSI letos několikrát potkali. Frameworky jsou speciálním druhem knihovny; dalo by se říct, taková “knihovna naruby”. Zatímco knihovny obvykle používáme tak, že je naimportujeme do svého programu a voláme jejich funkce a třídy; framework, poté co ho inicializujeme, naopak používá naše funkce a třídy, které mu poskytneme. Frameworky jsou tak trochu jako americký Hollywood nebo CIA: “Don’t call us, we call you”.

V případě Pytestu to funguje tak, že Pytest si najde naše testové funkce a ve vlastní režii je spouští a vypisuje výsledky. Od okamžiku, kdy spustíme Pytest, má kontrolu Pytest, a nikoli náš program. Můžeme říct, že náš program slouží Pytestu jako knihovna, kterou Pytest importuje a volá v ní testové funkce.

Proč jsem pro testování umocňovacího algoritmu vybral zrovna těchhle šest testovacích případů?

Okrajové případy a třídy ekvivalence vstupů

Při hledání důkazu korektnosti v nulté vlně jsme se snažili zformulovat argument, kterým doložíme, že algoritmus bude fungovat správně pro všechny vstupy (které splnují vstupní podmínku). Korektnost jsme dokazovali pomocí matematické indukce. Začali jsme tím, že jsme si ověřili, že algoritmus funguje na jednom konkrétním bázovém případu, x=0. Následně jsme zformulovali a dokázali indukční krok, tedy tvrzení že pokud algoritmus počítá správně pro nějaké k, pak vrátí správný výsledek i pro k+1. Když to dáme dohromady, máme dokázáno, že algoritmus funguje pro všechna x∈[0,∞).

Všimněme si, že prvním krokem důkazu pomocí matematické indukce je test s konkrétní hodnotou, abychom ustavili bázový případ.

Nyní bychom chtěli ověřit náš program čistě pomocí testů. Při testování hledáme několik reprezentativních vstupů takových, že když na nich náš program funguje, je pravděpodobné, že jsme jej napsali správně. Snažíme se vybrat vstupy, které nejsou komplikovanější než je nezbytně nutné, to proto, aby se o nich dalo dobře přemýšlet.

Jeden možný přístup k volbě testovacích případů je zamyslet se nad strukturou problému a nad strukturou programu, který problém řeší. Všechny přípustné vstupy si (s přihlédnutím k zadání problému a implementaci) můžeme rozdělit do tříd ekvivalence. Pokud se na nějakých dvou vstupech chová program rozdílně, bud proto, že to plyne ze zadání problému, nebo je to dáno implementací programu, budou tyto vstupy patřit do rozdílných tříd. Když si takto rozmyslíme třídy ekvivalence, z každé třídy vybereme jednoho reprezentanta a napíšeme pro něj test.

Pokud mezi vstupy existuje uspořádání nebo metrika, můžeme se navíc bavit o okrajových případech, což jsou vstupy, které leží u hranice se sousedními třídami. Tehdy má ještě smysl napsat testy pro tyto okrajové případy, protože se nám mohlo stát, že při programování jsme mezi nimi v kódu špatně rozlišili. Jeden nebo dva testy pro každou hranici by měly dostačovat.

Vztaženo na příklad s druhou mocninou, ze struktury problému nás napadne zkusit x = 0 a x = 1. Už proto, že pracujeme s celými čísly, a toto jsou význačné prvky (neutrální a jednotkový prvek). Navíc x = 0 je krajní hodnota povoleného rozsahu vstupů (jelikož náš algoritmus nefunguje pro záporné exponenty), a taky nepoučenému člověku nemusí být zřejmé, že 20=1.

Ze struktury implementace je vidět, že umocňovací program se bude chovat různě (bude provádět jiné příkazy) podle toho, jak dopadne podmínka v cyklu while. Chceme zkusit vstupy, aby se tělo cyklu while vůbec neprovedlo (x = 0), provedlo jednou (x = 1), a provedlo vícekrát (třeba x = 2). Tím vyzkoušíme všechny kvalitativně svébytné možnosti, jak může Python projít našim programem.

Chceme, aby každému, kdo se podívá na náš test, bylo jasné, co testovaná funkce má počítat. Pokud do testu zapíšeme testy až po x = 5, měl by v tom ty mocniny dvojky vidět na první pohled každý.

Má smysl zkusit nějaké velmi velké číslo? V Pythonu ne, protože inty v Pythonu mají neomezený rozsah, a protože takový test by trval zbytečně dlouho.

Co případy, kdy x není číslo, například x = "tučňák"? Funkce pow_two je otypovaná a přijímá pouze int, takže něco takového nemusíme ošetřovat, když máme typovou kontrolu. Typová kontrola je forma počítačem prováděného důkazu, a když máme na výběr, je lepší preferovat typy před testy všude tam, kde to rozumně jde.

Protože v Pythonu nemůžeme pomocí typu říct, že jako argument přijímáme jen nezáporné inty, má smysl přemýšlet nad testem pro záporný vstup. Samozřejmě nemá smysl psát test pro něco, co víme, že nefunguje; takže jen pokud jsme se rozhodli to řešit, můžeme napsat následující test, a příslušným způsobem opravit implementaci.

import pytest


def test_pow_two_negative() -> None:
    with pytest.raises(ValueError):
        pow_two(-1)

Pytest chrání mě i tebe

The purpose of […] tests is to confirm some property of the source files checked into the repository. […] One user writes a test to assert an invariant which they expect to be maintained. Other users execute the test later to check whether the invariant has been broken.

Bazel: Test encyclopedia (bazel.build)

Když napíšeš test, máš jistotu, že testový případ funguje jak má a že i v budoucnu bude fungovat jak má. Pokud na programu pracuje více lidí, nebo pokud se na programu pracuje delší dobu, je nezbytně nutné si tuto jistotu nějak zajistit. Provádět ručně důkaz korektnosti znovu po každé změně programu je nerealistické, zatímco pouštět testy pořád dokola funguje docela dobře, když jsou testy napsané tak, aby byly rychlé. Hezký pragmatický popis toho, jak a proč by testy měly patřit do každodenní programátorské praxe, najdeš v kapitolách 11 až 14 v knize Software Engineering at Google: Lessons Learned from Programming Over Time (Titus Winters a kol., 2020). Pokud se s ní někde potkáš, třeba v PDF, mohu jen dopořučit.

Pokročilá funkcionalita Pytestu

Funkcionalitu Pytestu můžeme rozšířit nainstalováním pluginů. Pluginy se instalují stejným způsobem, jako jakýkoliv jiný balíček určený pro programovací jazyk Python.

Timeout

Když nainstalujeme pytest-timeout, můžeme použít přepínač --timeout a říct tak Pytestu, kolik sekund může každý test nejdéle běžet. Hodí se to aby Pytest automaticky zabil zacyklené testy.

Barvičky

Když nainstalujeme pomocí pipu plugin pytest-sugar, bude výstup Pytestu barevnější a hezčejší.

Sofistikované chybové hlášky

Pytest sám o sobě vylepšuje chybové hlášky z assertů. Existuje plugin, který tyto chybové hlášky vylepší ještě více. Postačí nám pipem nainstalovat pytest-icdiff. (Existuje alternativní plugin se stejnou funkcionalitou nazvaný pytest-clarity. Pokud pro ně máš využití, nainstaluj si jeden z nich.)

HTML report

Pytestu můžeme říct, aby vygeneroval HTML stránku s reportem. Je potřeba nainstalovat plugin pytest-html, a následně použít přepínače --html=report.html --self-contained-html při spuštění pytestu.

Pokrytí

Pytest si během testování umí značit, které řádky tvého programu byly spuštěny. Obecně se dá říct, že dobré testy během svého provádění spustí velké procento řádků tvého programu. To, co se během testování vůbec nespustilo, se určitě neotestovalo. Naopak to ale neplatí; vysoké pokrytí ještě neznamená, že tvoje testy jsou dobře napsané.

Knihovna pro měření pokrytí v Pythonu se jmenuje Coverage.py, balíček v pipu se jmenuje coverage a plugin do pytestu je pytest-cov. Kromě řádkového pokrytí umí ještě měřit pokrytí větví, branch coverage.

python -m pip install pytest-cov
python -m pytest --cov=. --cov-branch --cov-report=html pow_two.py
firefox htmlcov/index.html

Červeně označené řádky nebyly během testování provedeny vůbec, žluté řádky částečně, přičemž vyskakující bublina vysvětluje, co chybělo k uplnému pokrytí větve, a zelené řádky jsou pokryté.

Na jaké pokrytí bychom měli při psaní testů cílit? “Eighty percent and no less!”, řekl Master Testivus.

Test discovery

pytest implements the following standard test discovery:

  • If no arguments are specified then collection starts from testpaths (if configured) or the current directory.
  • Alternatively, command line arguments can be used in any combination of directories, file names or node ids.
  • Recurse into directories, unless they match norecursedirs.
  • In those directories, search for test_*.py or *_test.py files, imported by their test package name.
  • From those files, collect test items:
    • test prefixed test functions or methods outside of class
    • test prefixed test functions or methods inside Test prefixed test classes (without an __init__ method)

Conventions for Python test discovery (pytest.org)

Právně uvedené nám říká, že pokud pustíme Pytest bez parametru, najde si Pytest všechny soubory v aktuální složce a podsložkách, jejichž jméno bud začíná na test_, nebo končí na _test.py a spustí v nich všechny funkce, jejichž jméno začíná na test_. Popsané chování nás navádí k tomu psát testy do zvláštních souborů, vedle souboru s kódem, který testujeme. Testovaný kód můžeš z testu importovat pomocí relativního importu, např. from .utils import pow_two.

Program ze zadání pochází z firmwaru přehrávače Zune a bug v něm je skutečný. V praxi se poprve projevil na konci roku 2008, což byl první přestupný rok po uvedení zařízení do prodeje.

Microsoft Zunes spontaneously dying all over the place (macworld.com, 31. 12. 2008)

Jediné místo, kde se program může zacyklit, je smyčka while ve funkci ConvertDays. Pokud funkci zavoláme na datum odpovídající poslednímu dni přestupného roku, tělo smyčky nijak nezmění hodnoty proměnných days a year. Takže se smyčka bude provádět pořád dokola, donekonečna.

The Zune Bug (Brian Hayes, 3. 1. 2009)

Pokud tě zarazilo, proč kod v zadání vypadá jak vypadá, tak je to proto, že se jedná o přepis z jazyka C do Pythonu. Původní leaknutý céčkový zdroják se nějakou dobu nacházel na http://pastie.org/349916. Pokud se na něj chceš podívat ted, můžeš použít The Wayback Machine na stránkách archive.org: rtc.c.

Zune bricking code uncovered: it’s a leap year mistake, and not Microsoft’s (guardian.com, 1. 1. 2009)

“We have as many testers as we have developers. And testers spend all their time testing, and developers spend half their time testing. We’re more of a testing, a quality software organization than we’re a software organization.”

— Bill Gates

Pytestový test, který vyzkouší, zda funkce ConvertDays funguje správně, a odhalí že ne, by mohl vypadat následujícím způsobem.

import datetime
import pytest


@pytest.mark.parametrize("days", range(1, 36891 + 1))
def test_ConvertDays(days):
    origin = datetime.date(1980, 1, 1)
    date = origin + datetime.timedelta(days=days - 1)

    assert ConvertDays(days) == (date.year, date.month, date.day)

Jak funkci ConvertDays opravit? Možností je víc; at už zkusíš cokoli, nezapomen si následně pustit testy.

V Pythonu dává asi největší smysl funkci smazat a použít pro manipulaci s datumy modul datetime ze standardní knihovny.
