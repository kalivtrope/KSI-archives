Zadání

Karlík řeší úlohu do KSI

Karlík takhle řešil jednu sice náročnou, ale zábavnou a naučnou úlohu do KSI. Měl ji uloženou v souboru uloha.py. Jakmile ji večer dokončil, zkopíroval si soubor s úlohou do souboru uloha_final.py. To proto, aby věděl, že takhle vypadá konečná verze. Potěšen svým úspěchem si šel za odměnu lehnout, přičemž si nechal zdát o možných vylepšeních úlohy. Některá z nich se mu líbila natolik, že hned ráno, jak se probudil, sedl k počítači a naprogramoval je. Bál se ale, aby si omylem nepřepsal nějakou část svého původního kódu, takže raději psal znovu do souboru uloha.py a až byl hotový, uložil si výsledek do souboru uloha_final_final.py, aby si svoje původní řešení nepřepsal, kdyby se k němu chtěl vrátit. Naráz ale dostal záblesk budoucnosti, ve kterém uviděl, kolik takhle souborů _final.py bude ještě potřebovat, a napadlo ho, jestli to nejde dělat nějak líp. Po chvíli hledání na internetu našel odpověď! Říká se jí verzovací systémy.

Verzovací systémy

Verzovací systémy jsou programy, které nám umožňují efektivně spravovat verze našeho kódu, takže se s nimi nemusíme bát editovat kód, jelikož se vždy můžeme vrátit k jeho starší verzi. A nejen že se můžeš vratit ke starší verzi, ale i se můžeš podívat, které řádky se mezi verzemi změnili. Dělají to tak, že sledují soubory, které se změnily a ty změny si ukládají.

To se hodí jak pro tvoje malé soukromé projekty, tak pro takové megaprojekty jako jádro Linuxu, do kterého touto formou přispělo přes 10 tisíc lidí. A proto se nyní podíváme na základní práci se snad nejrozšířenějším z nich - git.

Získáváme git

Git je primárně terminálový nástroj, nicméně existuje spousta GUI (Grafické Uživatelské Rozhraní/klikátko), které s ním umí pracovat. Získat samotný git není dnes nic složitého. Pokud jsi na systému založenému na Debianu, stačí ti jej doinstalovat přes sudo apt install git. Pokud jsi na Windows, můžeš si například stáhnout GitHub desktop, což je právě GUI nástroj, který je založený na snadnosti použití. Zároveň, pokud používáš nějaké IDE, je dost možné, že git máš již integrovaný v něm.

Jelikož hodně klientů vypadá jinak, budeme se v úloze zabývat převážně terminálovým rozhraním, přičemž jej budeme ve vhodné momenty doplňovat screenshoty z GitHub desktop.

Nastavujeme git

Pokud git používáme poprvé, musíme nastavit naše jméno a email. To proto, aby git poté věděl, komu má danou změnu přiřadit. Oba tyto údaje budou viditelné, pokud svůj repozitář někam nahraješ.

Toto provedeš pomocí následujících příkazů (Dolar na začátku znamená, že se jedná o příkaz. Tento znak ze svého příkazu proto vynechej.):

$ git config --global user.name "Tvoje Jmeno"
$ git config --global user.email "tvuj.email@example.com"

Zakládáme repozitář

Repozitář je složka, která se má sledovat pro změny. Sledují se všechny soubory v této složce i v podsložkách. Nový repozitář založíme tak, že přejdeme do složky, kterou chceme sledovat a spustíme příkaz git init:

$ cd muj_projekt/
$ git init
Initialized empty Git repository in /home/adam/muj_projekt/.git/

Když se nyní podíváš na výpis všech souborů (v Linuxu na terminálu ls -a, na Windows například Průzkumník souborů), všimneš si, že se v tvojí složce objevila nová složka pojmenovaná .git. Do této složky git ukládá všechnu historii a proto není dobrý nápad ji mazat.

V GitHub desktop dosáhneme podobného výsledku pomocí tlačítka Create new repository

Upravujeme náš kód

Nyní, když už máme vytvořený náš repozitář, je čas do něj přidat nějaké změny. Vytvořme si ve tvém oblíbeném editoru například program oblibene_jidlo.py, který ukládá naše oblíbené jídlo s následujícím kódem:

jidlo = input("Zadej svoje oblibene jidlo: ")

with open("oblibene_veci.txt", "w") as f:
    f.write(jidlo)

Nyní se pomocí git status zeptáme gitu, co se změnilo:

$ git status
On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)

    oblibene_jidlo.py

nothing added to commit but untracked files present (use "git add" to track)

Jak vidíš, git si všimnul, že jsme přidali náš soubor s programem. Nyní gitu řekneme, že chceme uložit změny našeho souboru do historie gitu. To provedeme pomocí příkazu git add.

$ git add oblibene_jidlo.py

Pokud bychom chtěly označit k uložení stavu všechny změněné soubory, místo jména souboru použijeme tečku: git add .. (V našem případě, kdy se změnil pouze jeden soubor, jsou tyto dva příkazy identické)

A abychom změny opravdu uložily, musíme provést tzv. commit, kterému předáme pomocí parametru -m zprávu, co jsme provedli:

$ git commit -m 'vytvoril jsem program'
[master (root-commit) 1983554] vytvoril jsem program
 1 file changed, 2 insertions(+)
 create mode 100644 oblibene_jidlo.py
Jakou verzi souboru vlastně git commit ukládá do historie?

Možná by tě napadlo, že při zavolání git commit se do historie uloží přesná kopie toho, co máš na disku. Schválně si to vyzkoušej! Uprav svůj soubor, zavolej git add soubor. Poté soubor znovu uprav a až poté zavolej git commit. Jakmile dokončíš tuto úlohu a budeš umět se podívat do historie commitů, zjistíš, že se neuložil nejnovější stav souboru, ale místo něj takový, jaký byl při volání git add.

Všechnu tuhle práci můžeme udělat i v GitHub desktop a to pomocí levého panelu, kde vybereme, co chceme commitnout, napíšeme zprávu a klikneme na Commit to main.

Provádíme další změny

Teď nám došlo, že asi nechceme ukládat oblíbené jídlo, pokud uživatel žádné nezadá. Modifikujme tedy náš program:

jidlo = input("Zadej svoje oblibene jidlo: ")
if not jidlo:
        exit()

with open("oblibene_veci.txt", "w") as f:
    f.write(jidlo)

A abychom otestovali, že funguje, tak si jej spustíme. Funguje? Výborně! Pojďme tedy uložit naše změny, to už umíme.

Nejdříve se podíváme, co se vlastně změnilo:

$ git status
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   oblibene_jidlo.py

Untracked files:
  (use "git add <file>..." to include in what will be committed)

    oblibene_veci.txt

no changes added to commit (use "git add" and/or "git commit -a")

Podívejme se na to! Přibyl nám další soubor oblibene_veci.txt. Ten ale asi nechceme ukládat, protože každý má jiné oblíbené jídlo, že? Chtělo by to tedy gitu nějak říci, že tenhle soubor má ignorovat. A přesně to uděláme pomocí souboru .gitignore, který vytvoříme a jako obsah mu dáme cestu k souboru, který se má ignorovat:

# ignoruj oblibene veci
oblibene_veci.txt
# popr. ignoruj vsechny .txt soubory
*.txt

Poté, co jsme tento soubor vytvořili, spusťme znovu git status:

$ git status
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   oblibene_jidlo.py

Untracked files:
  (use "git add <file>..." to include in what will be committed)

    .gitignore

no changes added to commit (use "git add" and/or "git commit -a")

Jupí! Zabralo to. Pojďme tedy uložit naše změny (teď už dvě, takže je raději rozdělíme do dvou commitů, v GUI vybrání pouze jednoho souboru provedeme odzaškrtnutím toho druhého):

$ git add .gitignore
$ git commit -m 'pridal jsem .gitignore'
$ git add oblibene_jidlo.py
$ git commit -m 'program neuklada jidlo, pokud neni zadano'

Díváme se do historie

Na to, abychom se podívali, co se změnilo v minulosti, slouží příkaz git log. Mnohem zajímavější ale pro nás bude překliknout se v GUI do záložky History, kde uvidíme seznam všech přechozích commitů. Z nich si můžeme rozkliknout jakýkoliv a podívat se, jak náš kód vypadal a co se změnilo.

Vracíme změny

Najednou ti došlo, že přeci jenom chceš uložit stav, kdy ti uživatel zadá prázdné jídlo. To proto, že chceš, aby měl možnost si svoje oblíbené jídlo smazat. Takže nyní máš dvě možnosti - buď manuálně znovu upravit kód tak, aby dělal to, co chceš, nebo vrátit zpět ten commit, ve kterém jsem toto přidali. V našem případě to není zas takový rozdíl, ale pokud bys měl komplikovanější kód, jistě rád využiješ možnosti vracet změny beze strachu, že jsi na nějakou zapomněl.

K tomu, abys v GitHub desktop vrátil nějaký commit, ti stačí v záložce History kliknout pravým tlačítkem na commit, který chceš vrátit a vybrat Revert commit. Tato akce ti automaticky vytvoří nový commit, který vrací zpět všechny změny z tvého vybraného. Vyzkoušej si to na náš commit program neuklada jidlo, pokud neni zadano a uvidíš sám, jak to funguje.

Pokud by tě zajímalo, jestli se něco podobného dá udělat v příkazové řádce, tak to určitě jde. K tomu slouží příkaz git revert, kterému předáš jako parametr hash commitu, který chceš vrátit. Ten můžeš zjistit pomocí git log, jakožto text hned za nápisem commit (kdyby ti git log vypsal například commit d4850a8c96e7bcd15fe65fa958bc9a7567cd9318, tak hash commitu je d4850a8c96e7bcd15fe65fa958bc9a7567cd9318, tudíž bys zavolal git revert d4850a8c96e7bcd15fe65fa958bc9a7567cd9318).

A máme hotovo

A je to! Už se umíme podívat na předchozí verze našich projektů a případně se v nich vracet zpět, takže se nemusíme strachovat editovat náš kód.

Řešení

Git Workflow

  1. Nejdříve musíme vytvořit repozitář pomocí git init
  2. Pak musíme označit soubor k uložení pomocí git add soubor
  3. A nyní již samotné uložení/commit pomocí git commit -m "zprava"
  4. A nakonec si vypíšeme změny příkazem git log

Kvíz

  • Jména/cesty k souborům, které chceme ignorovat, zapíšeme do souboru .gitignore
  • Soubory, které se změnily od posledního commitu vypíšeme pomocí git status
